# S-parks V6.5 功能同事工作流程

这份说明写给负责功能拓展的同事。

你的主要职责不是重做界面，而是在现有页面基础上把功能补完整、补稳定、补可扩展。

## 你主要负责什么

- 登录、角色、权限逻辑
- 功能流程拓展
- 后续数据结构
- 管理员、审核、账号体系相关能力

## 你尽量不要主动改什么

- 大面积视觉样式
- 页面整体排版
- 首页和个人主页的设计风格
- 无关文案润色
- 看起来“顺手优化”的 UI 重写

这些内容主要由 `logifore` 负责。

## 每次开始前固定流程

1. 先看一遍这 3 份文件：
   - `README.md`
   - `CONTRIBUTING.md`
   - `TEAM_WORKFLOW.md`
2. 确认这次只做 1 个功能目标。
3. 从最新 `main` 新建一个分支，不要直接在 `main` 上改。

推荐分支名：

- `feature/login-role-upgrade`
- `feature/admin-review-actions`
- `fix/auth-redirect-bug`

## 开发时只改必要文件

功能改动优先集中在：

- `assets/js/app.js`
- `assets/js/auth.js`
- `assets/js/router.js`
- `assets/js/config.js`
- `supabase/`

如果你发现必须改这些 UI 文件：

- `index.html`
- `assets/css/`
- `assets/js/renderers.js`
- `assets/js/content.js`

请遵守两条：

- 只做最小改动
- 在 PR 里明确写清楚为什么要改

## 绝对不要做的事

- 不要删除文件夹
- 不要删除旧版本
- 不要批量改文件名
- 不要因为“代码有点乱”就整仓库重构
- 不要把账号、密码、Token、密钥提交到仓库
- 不要直接把未确认代码推到 `main`

如果你看到旧代码像是没用了，也先不要删。先保留、备注、再让负责人确认。

## 每次功能开发的正确节奏

1. 先明确这次只解决什么问题。
2. 只改和这个问题直接相关的文件。
3. 改完先自己测试。
4. 确认没有误伤其他页面。
5. 提交 Pull Request，等待 review。

## 提交前自查清单

- 登录入口还能用
- 页面没有白屏
- 目标路由能正常进入
- 如果改了角色逻辑，至少检查：
  - 普通用户
  - 创作者
  - 管理员
- 如果改了页面结构，至少看一次桌面端效果
- 没有把无关文件一起提交

## Pull Request 里要写什么

至少写清楚这 4 件事：

- 你改了什么
- 为什么要改
- 影响了哪些页面或逻辑
- 你自己测了什么

如果你改了登录、角色、审核逻辑，最好把测试结果也写进去。

## 如果你不太会用 Codex

可以把任务交给 Codex，但要先让 Codex 读仓库里的：

- `AGENTS.md`
- `CONTRIBUTING.md`
- `TEAM_WORKFLOW.md`

然后你自己必须再做这几件事：

- 看一遍改动文件列表
- 看一遍核心 diff
- 确认没有误删文件
- 确认没有误改 UI 主结构

不要一键接受大面积自动修改。

## 如果出现这些情况，先停下来

- Codex 一次改了很多无关文件
- 你发现它动了大量 CSS
- 你发现它想删旧代码或旧目录
- 你不确定改动会不会影响主页视觉
- 你看不懂它为什么要改这么多

这时不要硬提交，先联系 `logifore` 一起看。

## 最稳妥的协作原则

- 一次只做一个功能
- 一次只提交一类改动
- 小步提交
- 不乱删
- 不乱改 UI
- 不直接碰 `main`

按这个流程走，项目基本就不会乱。
