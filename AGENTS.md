# S-parks V6.5 Agent Guide

本文件写给会在本仓库中工作的 Codex / AI 编码助手。

## 开工前必须先读

在开始分析、改代码、删文件、重构前，先读取这 3 份文件：

- `README.md`
- `CONTRIBUTING.md`
- `TEAM_WORKFLOW.md`

如果没有先读，不要直接开始改动。

## 当前协作分工

- `logifore` 负责：
  - UI 设计
  - 页面交互
  - Bug 修复
- 功能同事负责：
  - 功能拓展
  - 登录、角色、权限相关逻辑
  - 数据结构和后续接口接入

## 文件边界

优先把功能改动限制在这些文件：

- `assets/js/app.js`
- `assets/js/auth.js`
- `assets/js/router.js`
- `assets/js/config.js`
- `supabase/`

以下文件默认视为 UI / 展示层，除非任务明确要求，否则不要主动大改：

- `index.html`
- `assets/css/`
- `assets/js/renderers.js`
- `assets/js/content.js`

如果功能实现必须碰 UI 文件：

- 只做最小改动
- 不要顺手重写样式
- 不要顺手改文案风格
- 在最终说明里明确写出为什么必须改这些文件

## 安全规则

- 不要删除任何旧版本目录
- 不要删除仓库里的现有文档
- 不要批量重命名文件
- 不要因为“看起来没用”就删除旧代码，先标注并说明
- 不要覆盖他人最近已经完成的内容
- 不要把密钥、Token、账号密码、私密链接提交进仓库

## Git 工作方式

- 不要直接在 `main` 上开发
- 每次只做一个明确功能
- 先从最新 `main` 开分支
- 改动完成后先自查，再提 Pull Request

推荐分支命名：

- `feature/...`
- `fix/...`
- `docs/...`

## 修改策略

- 优先做增量式修改，不做整页推翻
- 优先做手术式修改，不顺手改无关内容
- 如果一个问题只需要改 1 个文件，不要扩散到 5 个文件
- 如果发现任务描述不清或会影响 UI 结构，先暂停并让负责人确认

## 自检要求

如果改动涉及登录或角色：

- 检查普通用户
- 检查创作者
- 检查管理员

如果改动涉及路由：

- 检查目标页面能否正常进入
- 检查返回路径和入口是否还在

如果改动涉及 UI：

- 至少看桌面端
- 至少看移动端

## 提交前必须确认

- 没有误删文件
- 没有误改无关页面
- 没有提交临时测试数据
- 没有提交敏感信息
- 改动说明里写清楚了影响范围

## 冲突处理

如果你发现下面任一情况，不要继续自动扩写：

- 任务会明显改到 UI 主结构
- 需要删除旧文件
- 需要重构多个核心文件
- 不确定这段代码是谁负责

此时应该先停下，并要求人工确认后再继续。
